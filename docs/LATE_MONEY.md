# late money（発走前オッズ時系列）収集・検証

公開の**確定オッズ**では EV>1.0 を達成できない（7手法×3年で実証、`RESEARCH_LOG.md`）。
残る唯一の未検証レバーが **late money** ＝「確定する“前”のオッズの動き」。締切直前に
特定馬が急に売れる（オッズ短縮）のは情報を持つ資金の兆候で、確定値には軌跡が残らない。

## なぜ前進収集が必須か
netkeiba は**過去レースの途中オッズを返さない**。よって既存DBでは検証不能で、
**これから発走するレースのオッズを発走前に自分で時系列取得して貯める**しかない。
バックテスト不可・蓄積待ちが本質。

## 仕組み

```
[開催日] poll-odds.yml (土日 JST 09:20 / 13:00 起動)
   └ scraper.poll_odds: 発走時刻を取得 → 各レースの発走前 60/30/15/10/5/3/1 分に
        単勝・複勝オッズをスナップショット → odds_snapshots(race_id,bet_type,poll_at,
        minutes_to_post,payload) に保存 → data-YYYY Release にアップロード
[蓄積後] evmodel.latemoney: 時系列から drift/late_drop を作り、
        「締切直前に売れた馬」戦略の回収率・CIを検証(確定結果と突合)
```

## 使い方（完全自動・電話操作不要）
- **自動**: `poll-odds.yml` が**土日** JST 09:00 と 13:00 に起動。まず「開催日判定」で
  発走時刻のあるレース数を確認し、**0（開催なし）なら即終了**（race_list 1回の軽いno-op）、
  **>0（開催あり）なら自動でポーリング**。レース当日に人が操作する必要は一切ない。
  （祝日月曜などの開催も拾いたくなったら cron に曜日を足すだけ。）
- **PROXY_URL**: netkeiba がランナーIPを弾く場合のみ Actions secret に設定（任意）。
- **手動（任意）**: Actions →「発走前オッズ収集(late money)」→ Run workflow → `date` 指定。
- **試験**: `python -m scraper.poll_odds --db test.db --race-id <id> --once`（即時1発）。

## 抽出特徴（`evmodel.latemoney`）
- `drift = log(late/early)`：負＝オッズ短縮＝売れた
- `late_drop = log(直近/中間)`：締切直前だけの急変（本命シグナル）
- 戦略 `steam`：drift が閾値以下（例 −0.05）の馬を単勝購入 → 実払戻で回収率・95%CI

## 検証と判定
1. 数週間で数百〜千レース蓄積 → `python -m evmodel.latemoney --db keiba_YYYY.db`
2. **CI下限>100%** が意味ある賭け数(≥100)で安定 → late money エッジ確認 → 実運用へ
3. 出なければ late money も却下（＝公開情報では勝てないという結論が確定）

## 課題・限界（正直な前提）
- **タイミング精度**: Actions cron は分刻みがずれる。開始後は相対時刻で `sleep` して撃つが、
  混雑や6時間上限で締切直前を外すことがある。精度を上げるなら常駐サーバが望ましい。
- **購入ウィンドウが極小**: 締切直前に観測→締切前に賭ける必要（実運用は投票API連携が前提）。
- **日本で有効な保証なし**: late money の +EV は欧米/香港の知見。JRA で残るかは未知。
- **ブロック/レート**: 開催日のみ・単勝複勝中心に抑える。exit 2 が出たら sleep を上げる。
